01. ROS 2 System Architecture

Pub/Sub, Services, Actions, Parameters, Nodes, Topics​

This chapter provides a comprehensive overview of the ROS 2 (Robot Operating System 2) system architecture, focusing on its core communication mechanisms and organizational concepts.

Introduction to ROS 2​

ROS 2 is the next generation of the Robot Operating System, designed to address the limitations of ROS 1 and meet the demands of modern robotics applications, including real-time systems, embedded platforms, and multi-robot deployments.

Why ROS 2?​

Improvements over ROS 1:

  • Real-time capabilities: Support for deterministic execution
  • Better security: Authentication, encryption, and access control
  • Multi-platform: Works on Linux, Windows, macOS
  • Production-ready: Designed for commercial deployments
  • DDS middleware: Industry-standard communication layer
  • Lifecycle management: Better control over node states
  • Composition: Run multiple nodes in a single process

Core Concepts​

Nodes​

Nodes are the fundamental building blocks of ROS 2 applications. Each node is an independent executable that performs a specific task.

Characteristics:

  • Single-purpose processes or threads
  • Can be written in Python, C++, or other supported languages
  • Communicate with other nodes through topics, services, and actions
  • Have lifecycle states (configuring, active, inactive, etc.)

Example:

import rclpy
from rclpy.node import Node

class MyNode(Node):
    def __init__(self):
        super().__init__('my_node')
        self.get_logger().info('Node initialized')

def main():
    rclpy.init()
    node = MyNode()
    rclpy.spin(node)
    rclpy.shutdown()

Topics (Publish-Subscribe)​

Topics implement the publish-subscribe pattern for continuous data streams. This is the most common communication method in ROS 2.

How it works:

  1. Publishers send messages to a topic
  2. Subscribers receive messages from a topic
  3. Many-to-many relationship: multiple publishers and subscribers per topic
  4. Asynchronous, decoupled communication

Use cases:

  • Sensor data streams (camera images, LiDAR scans)
  • Robot state information (position, velocity)
  • Continuous monitoring data

Example:

# Publisher
class TalkerNode(Node):
    def __init__(self):
        super().__init__('talker')
        self.publisher = self.create_publisher(String, 'chatter', 10)
        self.timer = self.create_timer(1.0, self.timer_callback)

    def timer_callback(self):
        msg = String()
        msg.data = 'Hello World'
        self.publisher.publish(msg)

# Subscriber
class ListenerNode(Node):
    def __init__(self):
        super().__init__('listener')
        self.subscription = self.create_subscription(
            String, 'chatter', self.listener_callback, 10)

    def listener_callback(self, msg):
        self.get_logger().info(f'Received: {msg.data}')

Quality of Service (QoS): ROS 2 allows fine-grained control over message delivery:

  • Reliability: Best effort vs. reliable delivery
  • History: Keep last N messages or all messages
  • Durability: Transient local (for late subscribers) vs. volatile
  • Lifespan: Message expiration time
  • Deadline: Expected publish rate

Services (Request-Response)​

Services provide synchronous request-response communication for one-off tasks.

How it works:

  1. Client sends a request
  2. Server processes the request
  3. Server sends back a response
  4. Client waits (blocking) for the response

Use cases:

  • One-time computations (inverse kinematics)
  • Configuration changes
  • State queries
  • Resource allocation

Example:

# Service Server
class AddTwoIntsServer(Node):
    def __init__(self):
        super().__init__('add_two_ints_server')
        self.srv = self.create_service(
            AddTwoInts, 'add_two_ints', self.add_callback)

    def add_callback(self, request, response):
        response.sum = request.a + request.b
        self.get_logger().info(f'{request.a} + {request.b} = {response.sum}')
        return response

# Service Client
class AddTwoIntsClient(Node):
    def __init__(self):
        super().__init__('add_two_ints_client')
        self.cli = self.create_client(AddTwoInts, 'add_two_ints')

    def send_request(self, a, b):
        request = AddTwoInts.Request()
        request.a = a
        request.b = b
        future = self.cli.call_async(request)
        return future

Actions (Long-Running Tasks)​

Actions are designed for tasks that take time to complete and may need to be cancelled or provide feedback.

How it works:

  1. Client sends a goal
  2. Server accepts or rejects the goal
  3. Server provides periodic feedback
  4. Client can cancel the goal at any time
  5. Server returns a final result

Three communication channels:

  • Goal: Client → Server (what to do)
  • Feedback: Server → Client (progress updates)
  • Result: Server → Client (final outcome)

Use cases:

  • Navigation to a waypoint
  • Grasping an object
  • Long computations
  • Motion planning

Example:

# Action Server
class FibonacciActionServer(Node):
    def __init__(self):
        super().__init__('fibonacci_server')
        self._action_server = ActionServer(
            self, Fibonacci, 'fibonacci', self.execute_callback)

    def execute_callback(self, goal_handle):
        feedback_msg = Fibonacci.Feedback()
        sequence = [0, 1]

        for i in range(1, goal_handle.request.order):
            # Check if cancelled
            if goal_handle.is_cancel_requested:
                goal_handle.canceled()
                return Fibonacci.Result()

            sequence.append(sequence[i] + sequence[i-1])
            feedback_msg.sequence = sequence
            goal_handle.publish_feedback(feedback_msg)
            time.sleep(1)

        goal_handle.succeed()
        result = Fibonacci.Result()
        result.sequence = sequence
        return result

Parameters​

Parameters are configuration values that can be set and modified at runtime.

Characteristics:

  • Stored per-node
  • Can be read and written dynamically
  • Support types: bool, int, float, string, arrays
  • Can be loaded from YAML files
  • Can be changed via command-line or programmatically

Example:

class ParameterNode(Node):
    def __init__(self):
        super().__init__('parameter_node')

        # Declare parameters with defaults
        self.declare_parameter('my_parameter', 'default_value')
        self.declare_parameter('max_speed', 1.0)

        # Read parameter
        param_value = self.get_parameter('my_parameter').value

        # Parameter callback for updates
        self.add_on_set_parameters_callback(self.parameter_callback)

    def parameter_callback(self, params):
        for param in params:
            if param.name == 'max_speed':
                self.get_logger().info(f'Max speed changed to {param.value}')
        return SetParametersResult(successful=True)

DDS Middleware​

ROS 2 uses DDS (Data Distribution Service) as its middleware layer, providing:

  • Automatic discovery: Nodes find each other without centralized master
  • QoS policies: Fine-grained control over communication
  • Security: Built-in authentication and encryption
  • Multiple vendors: Choose from different DDS implementations

Available DDS implementations:

  • FastDDS (default): eProsima's implementation
  • CycloneDDS: Eclipse project, lightweight
  • RTI Connext: Commercial, high-performance
  • Gurum DDS: Korean implementation

Best Practices​

When to Use Each Pattern​

Use Topics when:

  • Continuous data streams are needed
  • Multiple nodes need the same data
  • Decoupled, asynchronous communication is preferred
  • Examples: sensor data, robot state

Use Services when:

  • Quick, one-time operations
  • Synchronous response is needed
  • State queries or configuration
  • Examples: calculating IK, toggling modes

Use Actions when:

  • Long-running tasks
  • Progress feedback is important
  • Need ability to cancel
  • Examples: navigation, manipulation, planning

Node Design Principles​

  1. Single Responsibility: Each node should do one thing well
  2. Composability: Nodes should work together like UNIX tools
  3. Configurability: Use parameters for runtime configuration
  4. Observability: Publish diagnostic information
  5. Robustness: Handle failures and timeouts gracefully

Advanced Topics​

Lifecycle Nodes​

ROS 2 provides managed lifecycle nodes with states:

  • Unconfigured: Initial state
  • Inactive: Configured but not running
  • Active: Fully operational
  • Finalized: Shutdown

This allows better control over startup, shutdown, and error recovery.

Component Composition​

Multiple nodes can run in a single process for efficiency:

  • Reduced overhead
  • Zero-copy message passing within process
  • Easier debugging

Executors​

Executors control how callbacks are executed:

  • SingleThreadedExecutor: Default, simple
  • MultiThreadedExecutor: Parallel callback execution
  • StaticSingleThreadedExecutor: Optimized for real-time

Key Takeaways​

  • ROS 2 uses a distributed architecture with nodes communicating via middleware
  • Topics for continuous data streams (asynchronous, many-to-many)
  • Services for request-response patterns (synchronous, one-to-one)
  • Actions for long-running tasks with feedback and cancellation
  • Parameters for runtime configuration
  • DDS middleware provides discovery, QoS, and security
  • Choose the right communication pattern for your use case
